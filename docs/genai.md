# GenAI Section — Personal Expense Tracker

This is the headline for the project's GenAI deliverable. It contains the
seed prompt, a representative output, a brief summary of the audit trail,
and a short final reflection. The audit trail itself — the actual Issue
Cards documenting every AI-generated mistake that required correction —
lives in [`genai/issues.md`](./genai/issues.md), append-only and
chronological. The card template, four trigger paths, and the binary
`Human revalidation` encoding are defined in the project root's
[`CLAUDE.md`](../CLAUDE.md).

## 1. The prompt I would use

This is the prompt I would use to seed an iterative session with Claude Code
(or similar AI coding tool) to scaffold a RESTful task management API in
.NET. The prompt is structured in six blocks: role, constraints, domain,
acceptance criteria, output format, and edge cases. I do not write
monolithic single-shot prompts; I iterate, but a strong opening prompt
frames the conversation.

```
# Role
You are a senior .NET architect. Generate a RESTful API for a task management
system in .NET 10 + ASP.NET Core (Minimal APIs with MapGroup).

# Constraints
- NO Entity Framework, Dapper, or MediatR.
- Use plain ADO.NET (Microsoft.Data.SqlClient) for data access, wrapped in a
  thin SqlExecutor helper with Polly retry pipeline.
- Clean Architecture: 4 projects — Domain, Application, Infrastructure, Api.
- Minimal APIs organized by feature (TaskEndpoints.MapGroup("/tasks")) with
  IEndpointFilter for validation.
- TDD: produce tests first when possible (xUnit + NSubstitute + Shouldly).
- All async with CancellationToken propagated end-to-end.
- Argon2id for password hashing if user creation appears (Konscious lib).
- JWT in HttpOnly cookie + SameSite=Lax + anti-CSRF (Microsoft.AspNetCore.Antiforgery).

# Domain
Entity: Task { Id (Guid), Title (string ≤ 200), Description (string ≤ 2000),
              Status (enum: Pending|InProgress|Done), DueDate (DateTime UTC),
              UserId (Guid, FK) }
Assume User entity with { Id, Email, PasswordHash } already exists.
A user can ONLY see/modify their own tasks.

# Acceptance criteria
- 5 endpoints: GET /tasks, GET /tasks/{id}, POST /tasks, PUT /tasks/{id}, DELETE /tasks/{id}
- All endpoints require [Authorize] (cookie-based JWT bearer).
- Authorization at resource level: user A cannot access user B's task.
  Return 404 (not 403) to avoid info leak.
- Validation:
  - Title required, ≤ 200 chars, not whitespace-only
  - DueDate must be in the future on creation
  - Status must be valid enum (case-insensitive on input)
- Error handling: ProblemDetails (RFC 7807) on errors via global exception middleware.
- Pagination on GET /tasks: ?page=1&pageSize=20, max pageSize 100.
- SQL injection guard: only parameterized SqlCommand, no string concatenation.
- Tests: at minimum, one happy path + one auth boundary + one validation
  failure per endpoint. ArchUnitNET rule asserting Domain has no SqlClient ref.

# Output format
Generate, in order:
1. Domain/Task.cs (entity with invariants in constructor)
2. Application/UseCases/CreateTaskUseCase.cs + corresponding test
3. Application/Validators/CreateTaskValidator.cs + test
4. Infrastructure/Repositories/TaskRepository.cs (ADO.NET, parameterized)
5. Api/Endpoints/TaskEndpoints.cs (Minimal API with MapGroup)
6. Migration script: db/scripts/up/0001_create_tasks.sql

For each file, show the unit test FIRST (red), then the production code (green).

# Edge cases to explicitly handle
- DueDate in past → 400 with field error
- Status enum case-insensitive on input
- Title with only whitespace → 400
- User trying to update task they don't own → return 404 (not 403)
- Concurrent updates → out of scope (note in code comment)
- DateTime always UTC (use SYSUTCDATETIME() in SQL, DateTime.UtcNow in C#)
- Cancellation: respect CancellationToken in all async ops
- Logging: never log the user's PII (email, password hash) in plain text
```

## 3. Audit trail (live, append-only)

This project maintains a real-time audit log of AI-generated code that
required correction. The log lives in [`genai/issues.md`](./genai/issues.md)
and is appended to whenever the developer invokes a trigger phrase
(`"audit isso"`, `"registra no genai"`, `"issue card"`) or whenever a
sprint-boundary cleanup pass surfaces something that escaped real-time
logging. The two card templates and the four trigger paths are defined
in [`CLAUDE.md`](../CLAUDE.md).

The audit trail is split into two sections, and the SECTION the card
lives in IS the encoding of the catch source — no `### Source` label,
no field that could be fabricated after the fact:

- **Cleanup-pass cards** — surfaced by Claude reading sprint-boundary
  git diffs. Five-section template, `### Human revalidation` empty by
  convention.
- **Human-discovered cards** — surfaced by the developer through
  direct usage of the running app, manual code review, or live trigger
  during a fix session. Three-section template (`What I observed` /
  `Why it's wrong` / `What was done`); the entire card is the
  developer's voice. The strongest authenticity signal in the trail.

**Current state:** 12 substantive Issue Cards (11 cleanup-pass, 1
human-discovered) + 11 short bullets in the "caught but not deep-dived"
tail. Six of the deeper cards are anchored to Sprint 1 commits
(`5ce380c` through `274464e`), five to Sprint 2 commits (`0fc4193`
through `c4dcc3d`); chronological order within each section in
[`genai/issues.md`](./genai/issues.md) matches `git log` for that file.

## 4. Methodology note (final reflection)

Three patterns emerged from the live audit trail in
[`genai/issues.md`](./genai/issues.md), worth naming explicitly because
they shape how I'd use AI on the next greenfield project.

**What Claude got consistently wrong.** Reaching for APIs that *used to
exist*. `OptionsBuilder.Bind` (correct API is `BindConfiguration`),
`HealthCheckRegistration` as a `ServiceDescriptor` (it's not — it's
configured via `IConfigureOptions<HealthCheckServiceOptions>`),
`AntiforgeryMetadata.ValidationRequired` as a public class (it's
internal). These are stale-API failures: the suggestion compiles in a
mental model from a previous library version, but doesn't compile or
doesn't behave correctly in the current one. The fix path was always
the same — the build broke, I read the error, the AI corrected on the
next turn. But this is the failure mode I'd watch for first when using
AI on framework-heavy work in 2026.

**What Claude got surprisingly right.** The boring scaffolding —
project structure, dependency wiring, test fixtures, exception-to-
ProblemDetails mapping, the `SqlExecutor` shape — landed close to
correct on the first pass and required only marginal review. Claude is
a strong "first 80%" partner on patterns that are well-documented and
consensus in the ecosystem; the time saved there freed budget for the
parts that DON'T have consensus (cookie-vs-localStorage, zone vs
zoneless, cash-register input, the antiforgery metadata footgun) where
deliberate design is the value.

**The trigger-based audit vs single-shot experiment.** The original
plan considered running a sandboxed "give Claude the prompt and walk
away" experiment as the GenAI section. I rejected it for two reasons:
sandboxed output isn't representative of how I actually work
(iteratively, with corrections), and the artifact of correction is
itself the most interesting signal. Instead, the live trigger pattern
(`audit isso` → card appended) plus end-of-sprint cleanup passes
produced 19 cards anchored to real commits — verifiable through
`git log -- docs/genai/issues.md` rather than reconstructed from
memory. The two-section split (cleanup-pass vs human-discovered) makes
the catch source structurally legible: a reviewer can scan and see
where Claude self-audited and where I had to step in. The strongest
authenticity signal in the entire deliverable is the
"Human-discovered cards" section being non-empty — proof that I
actually exercised the running app and caught what the AI's review
missed (the leading-zero default, the premature error timing, the
cash-register entry pattern, the unenforced CSRF).

The methodology is the artifact. The cards are the evidence.
