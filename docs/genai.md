# GenAI Section — Task Management API Exercise

## Methodology

This document is maintained collaboratively by the developer and Claude Code. The
prompt and representative output are static (written once at the start of the
project). The Issue Cards are append-only entries written by Claude Code in
real-time during fix turns, triggered by explicit phrases ("audit isso",
"registra no genai") in user messages. The git history at `docs/genai.md`
confirms the chronology — the methodology is the artifact.

The project's `CLAUDE.md` file at the repository root contains the audit
directive that governs how this document is written.

## 1. The prompt I would use

This is the prompt I would use to seed an iterative session with Claude Code (or
similar AI coding tool) to scaffold a RESTful task management API in .NET. The
prompt is structured in six blocks: role, constraints, domain, acceptance
criteria, output format, and edge cases. I do not write monolithic single-shot
prompts; I iterate, but a strong opening prompt frames the conversation.

```
# Role
You are a senior .NET architect. Generate a RESTful API for a task management
system in .NET 10 + ASP.NET Core (Minimal APIs with MapGroup).

# Constraints
- NO Entity Framework, Dapper, or MediatR.
- Use plain ADO.NET (Microsoft.Data.SqlClient) for data access, wrapped in a
  thin SqlExecutor helper with Polly retry pipeline.
- Clean Architecture: 4 projects — Domain, Application, Infrastructure, Api.
- Minimal APIs organized by feature (TaskEndpoints.MapGroup("/tasks")) with
  IEndpointFilter for validation.
- TDD: produce tests first when possible (xUnit + NSubstitute + Shouldly).
- All async with CancellationToken propagated end-to-end.
- Argon2id for password hashing if user creation appears (Konscious lib).
- JWT in HttpOnly cookie + SameSite=Lax + anti-CSRF (Microsoft.AspNetCore.Antiforgery).

# Domain
Entity: Task { Id (Guid), Title (string ≤ 200), Description (string ≤ 2000),
              Status (enum: Pending|InProgress|Done), DueDate (DateTime UTC),
              UserId (Guid, FK) }
Assume User entity with { Id, Email, PasswordHash } already exists.
A user can ONLY see/modify their own tasks.

# Acceptance criteria
- 5 endpoints: GET /tasks, GET /tasks/{id}, POST /tasks, PUT /tasks/{id}, DELETE /tasks/{id}
- All endpoints require [Authorize] (cookie-based JWT bearer).
- Authorization at resource level: user A cannot access user B's task.
  Return 404 (not 403) to avoid info leak.
- Validation:
  - Title required, ≤ 200 chars, not whitespace-only
  - DueDate must be in the future on creation
  - Status must be valid enum (case-insensitive on input)
- Error handling: ProblemDetails (RFC 7807) on errors via global exception middleware.
- Pagination on GET /tasks: ?page=1&pageSize=20, max pageSize 100.
- SQL injection guard: only parameterized SqlCommand, no string concatenation.
- Tests: at minimum, one happy path + one auth boundary + one validation
  failure per endpoint. ArchUnitNET rule asserting Domain has no SqlClient ref.

# Output format
Generate, in order:
1. Domain/Task.cs (entity with invariants in constructor)
2. Application/UseCases/CreateTaskUseCase.cs + corresponding test
3. Application/Validators/CreateTaskValidator.cs + test
4. Infrastructure/Repositories/TaskRepository.cs (ADO.NET, parameterized)
5. Api/Endpoints/TaskEndpoints.cs (Minimal API with MapGroup)
6. Migration script: db/scripts/up/0001_create_tasks.sql

For each file, show the unit test FIRST (red), then the production code (green).

# Edge cases to explicitly handle
- DueDate in past → 400 with field error
- Status enum case-insensitive on input
- Title with only whitespace → 400
- User trying to update task they don't own → return 404 (not 403)
- Concurrent updates → out of scope (note in code comment)
- DateTime always UTC (use SYSUTCDATETIME() in SQL, DateTime.UtcNow in C#)
- Cancellation: respect CancellationToken in all async ops
- Logging: never log the user's PII (email, password hash) in plain text
```
